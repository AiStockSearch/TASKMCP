# Архитектура: пять слоёв (заказчик → LLM → Vault → MCP → Cursor)

Этот документ фиксирует **целевую** сквозную схему: от сбора требований до исполнения в IDE. Ниже — исходная формулировка ролей и **привязка к артефактам этого репозитория**.

---

## Слой 0 — сбор требований (LangChain Interviewer)

Агент с системным промптом «Architect-Inquisitor» не пускает задачу в работу, пока не вытащит из заказчика стек, критерии готовности, ограничения по latency и затронутые файлы. Результат — одобренный JSON, который ложится в таблицу **`requirements`** (в т.ч. поле **`spec_json`**).

## Слой 1 — рассуждение (IJ / мощная LLM)

Отдельный инстанс модели берёт утверждённое ТЗ и делает декомпозицию на атомарные задачи. Он не пишет код — только проектирует. Выход — структура вида **`tasks_decomposition`** (в общем JSON/planning payload), далее нормализуемый в **`tasks`** и связи **`task_files`**.

## Слой 2 — хранилище (PostgreSQL «The Vault»)

Ядро данных:

| Таблица | Назначение |
|---------|------------|
| **`requirements`** | Исходник + **`spec_json`** |
| **`tasks`** | Статус, приоритет, привязки к requirement/epic |
| **`task_files`** | Какой путь к какой задаче |

Оркестратор (например LangChain Planner) после слоёв 0–1 применяет **идемпотентные** `INSERT`/`UPSERT` по контракту [`writer-contract.md`](writer-contract.md). Пакетный формат примера: [`examples/planbundle_rn_auth.json`](../examples/planbundle_rn_auth.json).

Путь **асинхронной доставки** в Vault в этом репо: [`orchestrator/`](../orchestrator/README.md) — вебхук → **Redis** → воркер **arq** применяет тот же transactional write-side.

**Внешняя память:** Cursor не обязан держать весь контекст в голове — состояние бэклога и спецификации живёт в БД (и при необходимости в Memory Bank / KB).

## Слой 3 — MCP-сервер на Go (Vault Bridge)

Тонкий мост Cursor ↔ Postgres: инструменты чтения/записи бэклога и смежных подсистем.

**Минимальный набор под исполнение (концепция «три опоры»):**

| Роль в схеме | Реальные MCP tools в репозитории |
|----------------|----------------------------------|
| Взять следующую задачу + файлы | **`get_next_task`** (+ при необходимости **`get_task`**, **`list_tasks`**) |
| Закрыть с отчётом | **`complete_task`** |
| Поиск контекста | Не отдельное имя `search_context` — использовать **`kb_hybrid_search`** / **`kb_search_context`** по базе знаний и/или выборки задач через **`list_tasks`** |

Полный перечень см. в [`README.md`](../README.md) (раздел MCP tools): эпики, GitHub, Memory Bank (`mb_*`), и т.д.

## Слой 4 — Cursor как исполнитель

Не получает расплывчатых целей как единственный вход: ответы **`get_next_task`** и **`get_task`** содержат поля задачи плюс, если у задачи задан **`requirement_id`**, — **`requirement_title`** и **`spec_json`** из таблицы **`requirements`** (критерии готовности, ограничения, структура ТЗ). Отдельный MCP-tool «только requirement» не требуется для исполнения: источник правды — Vault, не дублирование в Memory Bank. **`mb_*`** и KB по-прежнему для правил проекта, планов и RAG поверх того же репозитория. После работы вызывается **`complete_task`** с отчётом.

## Роли участников

| Участник | Роль |
|----------|------|
| **Ты** | Заказчик: утверждаешь требования и приоритеты |
| **LangChain / LLM** | Архитектор и менеджер бэклога: интервью, декомпозиция, запись в Vault (напрямую или через оркестратор) |
| **Cursor** | Исполнитель с чётким ТЗ из Vault через MCP |

**Главный эффект:** контекстное окно агента в Cursor тратится на исполнение и правки, а не на бесконечное «раскручивание» задачи с нуля — чертёж уже подготовлен слоями 0–2.
