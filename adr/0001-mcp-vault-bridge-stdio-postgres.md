## Контекст

Строится автономный пайплайн разработки (“Cursor‑Rolls‑Royce”), где PostgreSQL (“The Vault”) хранит требования и декомпозированные задачи:

- `requirements(id, title, spec_json, status)`
- `tasks(id, requirement_id, title, description, status, priority)` со статусами `todo | in_progress | done`
- `task_files(id, task_id, file_path)`

Нужно дать Cursor MCP‑интеграцию по **stdio** (локальный subprocess), чтобы:

- выдавать следующую задачу на выполнение,
- возвращать список связанных файлов,
- принимать отчёт выполнения и помечать задачу завершённой,
- при необходимости привязывать новые файлы к задаче.

Система подразумевает конкурентный запуск нескольких агентов, поэтому выдача “следующей задачи” должна быть атомарной и безопасной от гонок.

## Решение

Реализовать отдельный MCP Server на Go (“Vault Bridge”), работающий по **stdio transport**, который подключается к Postgres по `DATABASE_URL` и предоставляет tools:

- `get_next_task`: внутри транзакции выбирает одну задачу `status='todo'` с минимальным `priority`, используя `FOR UPDATE OF t SKIP LOCKED` (блокируется только строка `tasks`), переводит задачу в `in_progress`, возвращает структурированный ответ с `file_paths` и, при `requirement_id`, полями `requirement_title` / `spec_json` из `requirements`.
- `complete_task`: строго завершает только `in_progress` задачу, переводит в `done`, дописывает Execution Report в `description`.
- `add_context_file`: идемпотентно добавляет связку `(task_id, file_path)` в `task_files`.

Сервер не должен падать при временно недоступной БД: он стартует, логирует предупреждение и возвращает понятные ошибки из tools до восстановления доступности.

## Рассмотренные варианты

### Вариант A — `github.com/mark3labs/mcp-go` + `server.ServeStdio`

- Плюсы:
  - Хорошо документированный stdio transport.
  - Простой контракт tools/handlers.
  - Подходит для “тонкого” bridge‑сервиса без лишней магии.
- Минусы:
  - Требует ручного описания схемы параметров tools (что в данном случае приемлемо).

### Вариант B — `github.com/metoro-io/mcp-golang` stdio transport

- Плюсы:
  - Удобные struct‑аргументы с автогенерацией JSON Schema.
- Минусы:
  - Меньше “де‑факто стандартизации” вокруг Cursor‑ориентированных примеров (по сравнению с вариантом A).

### Вариант C — HTTP сервис + отдельный MCP прокси

- Плюсы:
  - Унификация транспорта, проще деплой в кластер.
- Минусы:
  - Лишняя прокладка для stdio‑ориентированного Cursor.
  - Больше подвижных частей (прокси, auth, сеть), хуже DX на старте.

## Почему так

- Cursor ожидает MCP как subprocess по stdio: минимальная латентность и простая эксплуатация.
- Postgres уже является источником правды (“Vault”), поэтому bridge должен быть максимально прямолинейным: транзакции, `SKIP LOCKED`, явные ошибки.
- Устойчивость: даже при неверном `DATABASE_URL` MCP‑сервер остаётся поднятым (Cursor не теряет “server offerings”), а tools возвращают диагностируемые ошибки.

## Последствия

- **Плюсы**:
  - Конкурентно‑безопасная раздача задач нескольким агентам.
  - Простая модель владения: одна бинарь + `DATABASE_URL`.
  - Минимальный surface area для отказов.
- **Минусы**:
  - Дальнейшие возможности (отдельный `get_requirement`, heartbeat, и т.д.) по-прежнему расширяют контракт tools; `spec_json` для исполнителя уже отдаётся в `get_next_task` / `get_task` при связанной задаче.
  - Идемпотентность `add_context_file` лучше фиксировать ограничением UNIQUE в БД (опциональное улучшение).

## Примечания по данным и миграциям (рекомендации)

- Индексы:
  - `tasks(status, priority, id)` для быстрого выбора следующей задачи.
  - `task_files(task_id, file_path)`; желательно UNIQUE(task_id, file_path).
- Для UUID:
  - Генерировать UUID в приложении (Go), не полагаясь на `gen_random_uuid()` и расширения Postgres.
